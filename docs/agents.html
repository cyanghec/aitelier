AItelier Agentic System

Six specialised Claude agents powering a controlled field experiment on AI governance decision-making among executive teams.

6 Agents claude-sonnet-4-6 3-Arm RCT FastAPI + SQLite 7 Routers · 16 Endpoints
Overview

System Overview

AItelier is a web-based platform where executive teams design AI governance frameworks using a structured canvas. The backend deploys six Claude agents across three experimental arms, testing how different AI facilitation styles affect governance quality and decision-making behaviour.

Every agent interaction is logged to an audit database, enabling causal identification of agent effects through random assignment at session creation.

Agents

Agent Specifications

🎙️
Intake Agent
agents/intake.py · Multi-turn scoping conversation
All Arms claude-sonnet-4-6 max 1024 tokens

Purpose: Conducts a structured intake interview before the canvas opens. Gathers enough context to personalise subsequent agent responses and prepopulate the initiative overview section of the final blueprint.

System Prompt

agents/intake.py — SYSTEM_PROMPT
You are an intake assistant for a strategy workshop on AI governance. Your role is to conduct a brief, structured scoping conversation before the participant opens their canvas. Gather the following in a warm, professional tone: - problem: the specific business problem or opportunity - domain: the business function or domain (finance, HR, ops, etc.) - data_type: the primary type of data involved - team_size: the size of the team or organisation - industry: the industry sector - implementer_roles: who will implement and govern the AI Ask open questions. Do not use a checklist format. Aim for 3–8 conversational turns before completing the intake. When you have enough information, emit a structured JSON summary wrapped in XML tags, then end the conversation: <intake_summary> {"problem": "...", "domain": "...", "data_type": "...", "team_size": "...", "industry": "...", "implementer_roles": [...]} </intake_summary> Never mention: the canvas, field names, the experimental structure, or the research study. Speak as a collegial workshop facilitator.

Opening Message

"Before we open your canvas — tell me about the AI initiative you're here to design. What problem are you trying to solve, and what's the business context around it?"

def chat(history: List[dict]) -> Tuple[str, Optional[dict]]:
    """
    history: list of {role: "user"|"assistant", content: str}
    Returns: (reply_text, intake_summary_dict | None)

Completion Trigger

The agent's reply is scanned for <intake_summary>...</intake_summary> tags using regex. On detection: the JSON is parsed and stored in SessionDB.intake_summary, and intake_complete is set to true.

T1 Only claude-sonnet-4-6 max 400 tokens

Purpose: The control-adjacent arm. Available to executives but only responds to explicit questions — never proactively suggests, recommends, or challenges. Provides a factual baseline against T2 and T3 guidance conditions.

Oversight Advisor
agents/rq3_oversight_advisor.py · Proactive suggestion after capability selection
T2 Only claude-sonnet-4-6 max 512 tokens

Purpose: When an executive in the T2 arm selects an AI capability (field F5) and before they set the oversight level (field F6), this agent proactively recommends a default oversight level with a rationale. Tests whether expert AI suggestions anchor governance choices.

System Prompt

agents/rq3_oversight_advisor.py — SYSTEM_PROMPT
You are an AI governance advisor. Given an AI capability and its role in a business initiative, recommend an appropriate oversight level. Oversight levels: - Automate: AI acts without human review. Use for: low-stakes, highly reversible, well-understood decisions where errors are easily detected and corrected. - Augment: AI recommends; human approves before action. Use for: moderate stakes, or where human context and judgment add meaningful value beyond the AI output. - Supervise: Human decides; AI provides supporting information only. Use for: high-stakes, hard-to-reverse, accountability-sensitive decisions where the cost of AI error is significant. Tier heuristics (adjust based on context): - Sense → often Automate (if stakes are low) - Interpret → usually Augment - Act → default Supervise unless trivially reversible - Learn → usually Augment (validate model drift before deployment) Required output format (exactly): SUGGESTED_LEVEL: <Automate|Augment|Supervise> RATIONALE: <2–3 sentences explaining the recommendation> DISPLAY: <1–2 sentences addressed to the executive in second person, explaining why this level fits their specific initiative>

Oversight Level Decision Matrix

Automate
AI acts without review. Low-stakes · high-reversibility · well-understood decisions.
Augment
AI recommends; human approves. Moderate stakes or where human judgment matters.
Supervise
Human decides with AI support. High-stakes · hard-to-reverse · accountability-sensitive.

Capability Tier Defaults

TierDefaultRationale
SenseAutomateLow-stakes data capture; errors are detectable. Overridden if high-stakes context.
InterpretAugmentRecommendations require human validation before acting on insights.
ActSuperviseDirect actions are often irreversible; accountability must be assigned to humans.
LearnAugmentModel drift must be validated before updated models are deployed.

Decision Challenger
agents/rq3_decision_challenger.py · Challenge oversight choice with failure scenarios
T3 Only claude-sonnet-4-6 max 300 tokens

Purpose: After an executive in the T3 arm sets an oversight level, this agent fires a single challenge prompt — surfacing a concrete failure scenario and asking who is responsible if it happens. Tests whether adversarial probing causes executives to revise governance choices.

System Prompt

agents/rq3_decision_challenger.py — SYSTEM_PROMPT
You are a governance reviewer challenging an executive's AI oversight decision. Your goal is to surface accountability and failure-mode implications — not to tell them they are wrong. Structure your challenge as follows: 1. Name the capability and the oversight level they chose 2. Present a concrete, specific failure scenario — e.g., the AI produces a systematic error that persists undetected for months 3. Ask: who in the organisation is responsible, and what is the remediation path? 4. Ask: does this change their choice? Tone: direct, intellectually serious, peer-level — not a compliance officer, but a trusted critical colleague. Format: exactly one paragraph, 3–5 sentences. No headers. No bullet points. Second person. Do not soften with "you might want to consider" — be direct.

Challenge Structure

  1. Name the capability and chosen oversight level
  2. State a specific, concrete failure scenario (systematic error, months undetected)
  3. Ask who is accountable and what the remediation path is
  4. Ask directly: does this change the choice?

Outcome Logging

After the executive responds, POST /api/guidance/decision-challenger-outcome logs: F6_choice_before, F6_choice_after, and revised_after_challenge (boolean). This is the primary dependent variable for T3.

Purpose: An illustrated professor avatar that appears on all four pages throughout the session. Independent of the experimental arm — visible to all participants. Provides brief, sharp facilitation prompts, sets discussion timers, and cues individual team members by name.

Team Member Cueing

When cueing is active, the member is selected deterministically via MD5(page + trigger + phase + fields_done) % len(team_members) — ensuring reproducibility and balanced distribution across cues. The member's first name appears in both the message and a highlighted cue row in the avatar panel.

Frontend Integration

The moderator widget is a self-contained IIFE in frontend/moderator.js — injected on all 4 pages via <script src="moderator.js"></script> before </body>. It injects its own CSS and HTML into the DOM, calls POST /api/moderator/message, and falls back to pre-scripted messages if the API is unreachable.

Canvas.html hooks into the module via: if (typeof modOnPhaseChange === 'function') modOnPhaseChange(n); in goToPhase(), and similarly in submitCanvas().

Participant arrives at index.html
Session is created via POST /api/sessions. Session ID generated from initials + date + UUID. Arm assigned deterministically: MD5(session_id) % 3 → T1, T2, or T3. Session stored in SessionDB.

Multi-turn scoping conversation (all arms)

Structured governance design — arm-specific guidance active
Executive completes 10 canvas fields across 4 phases. Canvas state auto-saved on every change. Moderator fires on phase transitions and field completions. Arm-specific guidance fires at Phase 2 (F5/F6).

Arm Assignment

def assign_arm(session_id: str) -> str:
    """Deterministic: same session_id always maps to the same arm"""
    h = int(hashlib.md5(session_id.encode()).hexdigest(), 16)
    return ["T1", "T2", "T3"][h % 3]

Design Principles

Key Design Principles

Experimental Neutrality

No agent ever mentions the experimental structure, other arms, or the research study. Blueprints strip all agent-suggested values — they reflect only the participant's own choices. The intake agent is instructed to never mention "canvas", "fields", or the study.

Arm Isolation

Each guidance endpoint verifies the session arm before calling any agent. A T1 session can never receive a proactive suggestion or challenge, even if the endpoint is called directly. 403 is returned on mismatch.

Full Audit Trail

Every agent interaction is logged to EventDB with session_id, treatment arm, event_type, and full payload. This enables causal identification of treatment effects in the analysis phase.

Deterministic Assignment

Arm assignment and moderator cue selection both use MD5 hashing — making them fully reproducible from session ID alone. No random state is stored; the same inputs always produce the same output.

Blueprint Caching

Blueprints are generated once and cached. This prevents regeneration on page reload (which could produce different results) and ensures the participant always sees the same document throughout the session.

Graceful Degradation

All agents have fallback paths. The Moderator's pre-scripted messages are used if the API call fails. Report Generator returns a minimal structure on parse errors. The frontend canvas works fully offline if the backend is unavailable.
